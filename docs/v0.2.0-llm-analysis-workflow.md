# Spec: recodex 0.2.0 LLM Analysis Workflow

## Objective

Build an evidence-grounded LLM workflow for AI coding transcript analysis. The system must show how the LLM processed the session instead of sending raw chat logs into one prompt.

The user should be able to run a workflow analysis from the dashboard report area and see:

- normalized trace, not raw chat logs
- deterministic transcript facts and episode segmentation
- evidence packs sent to the LLM
- LLM extraction results per episode
- clustered cross-episode patterns
- validation status for each finding
- report synthesis, gated skill candidates, and feedback loop records

## Tech Stack

- Python 3.10+ backend under `src/recodex`
- SQLite persistence through `src/recodex/db.py`
- Existing LLM provider abstraction in `src/recodex/llm.py`
- React + Vite dashboard under `dashboard/`
- Python `unittest` tests under `tests/`

## Commands

- Backend tests: `PYTHONPATH=src python3 -m unittest discover -s tests`
- Focused workflow tests: `PYTHONPATH=src python3 -m unittest tests.test_llm_workflow tests.test_dashboard_server`
- Dashboard build: `npm run build` from `dashboard/`
- Local dashboard: `python3 -m recodex serve --dashboard-dir dashboard/dist --port 8000`

- `src/recodex/analysis_workflow.py` -> deterministic segmentation, evidence packs, workflow orchestration
- `docs/v0.2.0-transcript-graph-lineage.md` -> durable transcript graph and lineage contract

## Code Style

Use typed dataclasses for internal workflow data and plain dictionaries at API boundaries.

```python
@dataclass(frozen=True)
class WorkflowStageResult:
    stage: str
    status: str
    input_summary: dict[str, object]
    output: dict[str, object]
    warnings: tuple[str, ...] = ()
```

Boundary functions return JSON-serializable dictionaries. LLM outputs are treated as untrusted data and validated before persistence or rendering.

## Workflow

1. **Normalize Trace**
   Convert Codex, Cursor, and Claude Code records into a shared structure:
   `session -> turn -> event -> tool_call/result -> file/path/test/error/user_correction`.
   The workflow must not analyze raw transcript text directly.
   The durable design for this layer is specified in
   `docs/v0.2.0-transcript-graph-lineage.md`.

2. **Episode Segmentation**
   Split by task phase instead of token count:
   `user_request`, `planning`, `tool_execution`, `failure_retry`, `patch`,
   `verification`, `user_correction`, `final_response`.

3. **Deterministic Pre-pass**
   Extract facts in code before any LLM call:
   command failures, test runs, touched files, sandbox/network errors, user corrections,
   repeated file reads, and skipped verification.

4. **Evidence Pack**
   Build a small pack per episode:
   summary, source refs, bounded raw excerpts, command/status records, and file refs.
   LLM stages can only use evidence packs and normalized facts.

5. **LLM Issue Extraction**
   For each evidence pack, output structured JSON:
   `issue_type`, `severity`, `evidence_refs`, `user_impact`,
   `root_cause_hypothesis`, `recommended_change`, `confidence`, `missing_evidence`.

6. **Cluster Patterns**
   Merge issues into patterns such as:
   not reading code before implementation, test failure without root-cause analysis,
   unstable LLM JSON output, and importer performance bottlenecks.
   This follows a per-instance extraction plus clustering shape similar to
   LLM-as-qualitative-judge workflows.

7. **Validation Pass**
   A second pass only reviews support:
   whether the issue is evidence-backed, whether it cites turn/file/command refs,
   and whether a one-off issue is being overgeneralized into a reusable skill.
   Low-confidence items go to a human queue.

8. **Report Synthesis**
   The report page shows only validated clusters:
   problem, evidence, impact, recommendation, and possible artifact.
   It must not show the HTML report or full LLM raw prose.

9. **Skill Drafting**
   Generate `SKILL.md` only when a pattern is repeated, evidence-backed,
   executable, and human-confirmed. Otherwise keep it as a report recommendation.

10. **Feedback Loop**
    User accept/reject writes back to:
    `analysis_examples`, `judge_failures`, and `prompt_versions`.
    These records can become recodex's own eval dataset later.

## LLM Responsibilities

- Extractor: identify concrete issues from one bounded evidence pack.
- Clusterer: merge repeated issues into stable patterns and artifact candidates.
- Validator: reject unsupported findings, lower confidence when evidence is weak, and route low-confidence items to human review.
- Reporter: create a short report using only validated clusters.

Deterministic code is responsible for file discovery, parsing, segmentation, redaction, token budgeting, evidence IDs, cache keys, and API errors.

## Testing Strategy

- Unit tests verify deterministic segmentation and evidence IDs.
- Fake LLM tests verify all workflow stages run without network.
- API tests verify `/analysis/run` with `mode=workflow`.
- Existing report and LLM tests must continue passing.
- Dashboard build verifies TypeScript and UI integration.

## Boundaries

- Always: redact evidence excerpts before LLM calls.
- Always: every LLM finding must cite evidence IDs.
- Always: report validation errors in structured API responses.
- Ask first: adding new third-party dependencies or changing provider defaults.
- Never: send cloud LLM requests while local-only mode blocks cloud calls.
- Never: render raw full transcript as the analysis report.

## Success Criteria

- A user can run workflow analysis for a selected session from the dashboard.
- The API returns stage-level processing details.
- The UI shows what each LLM stage processed and produced.
- Mock workflow analysis runs offline and writes a report JSON.
- Existing report generation and improvement proposal flows still work.
