# Spec: recodex 0.2.0 Transcript Graph and Lineage

## Objective

Design a durable structured transcript layer for Codex, Cursor, Claude Code, and future AI coding tools.

This layer is the source of truth for LLM analysis. LLM workflows should consume normalized transcript graph records and evidence packs, not raw chat text.

Success means recodex can answer:

- What session did this event come from?
- Which user request, assistant response, tool call, tool result, file path, test result, and error belong together?
- Which source file and raw row produced a normalized event?
- Which events caused a later issue, cluster, report, or skill recommendation?
- Which imported records are new, updated, duplicated, or superseded?

## Current Gap

The current importer normalizes tool records into `SessionRecord` and `TranscriptEvent`.
That is good enough for search and basic reports, but too thin for reliable LLM analysis:

- event text mixes metadata, command, output, and content
- turn boundaries are inferred late in `analysis_workflow.py`
- tool calls and tool results are not first-class linked records
- file refs, test refs, and error refs are not persisted as structured facts
- provenance points to a transcript file, not a stable raw row or source span
- downstream reports cannot explain the full chain from raw row to issue to cluster to skill

## Design Principle

Use a transcript graph, not a flat event list.

The graph is append-friendly, source-preserving, and evidence-addressable:

```text
raw_artifact
  -> raw_record
  -> session
  -> turn
  -> event
  -> tool_call/tool_result
  -> file_ref/test_ref/error_ref/user_correction
  -> episode
  -> evidence_pack
  -> issue
  -> cluster
  -> validation
  -> report/skill_candidate
```

## Core Objects

### RawArtifact

Represents a source file, SQLite row group, log file, or imported directory item.

Fields:

- `artifact_id`: stable hash of source type, path, inode/mtime when available
- `source_type`: `codex`, `cursor`, `claude-code`, `openclew`, `hermes`, `manual`
- `source_path`: absolute path or logical source URI
- `content_hash`: full content hash or provider row hash
- `mtime`, `size_bytes`
- `ingest_run_id`
- `first_seen_at`, `last_seen_at`

### RawRecord

Represents one original row/message/log entry before normalization.

Fields:

- `raw_record_id`: stable hash of artifact id + provider row id/index + raw hash
- `artifact_id`
- `provider_record_id`: original UUID/request ID/row key when available
- `physical_index`: JSONL line, SQLite row ordinal, or log block index
- `raw_json`: compact JSON when structured
- `raw_text_preview`
- `raw_hash`
- `created_at`

### Session

Represents a tool conversation or coding session.

Fields:

- `session_id`
- `source_type`
- `title`
- `project_path`
- `workspace_id`
- `model`
- `started_at`, `ended_at`, `updated_at`
- `status`
- `raw_artifact_ids`

### Turn

Represents one user-visible interaction cycle.

Fields:

- `turn_id`
- `session_id`
- `turn_index`
- `initiator`: `user`, `assistant`, `system`, `tool`
- `phase_hint`
- `started_at`, `ended_at`
- `parent_turn_id`
- `source_refs`

### Event

Represents one normalized semantic event.

Fields:

- `event_id`
- `session_id`
- `turn_id`
- `event_index`
- `role`: `user`, `assistant`, `tool`, `system`, `developer`, `unknown`
- `event_type`: `message`, `tool_call`, `tool_result`, `patch`, `file_read`, `test_run`, `error`, `user_correction`, `final_response`
- `phase`: `user_request`, `planning`, `tool_execution`, `failure_retry`, `patch`, `verification`, `user_correction`, `final_response`
- `created_at`
- `source_ref`
- `text_excerpt`
- `raw_record_ids`
- `metadata_json`

### ToolCall

Represents a tool invocation, even when the source tool stores it inside a message.

Fields:

- `tool_call_id`
- `event_id`
- `turn_id`
- `tool_name`
- `command`
- `arguments_json`
- `cwd`
- `started_at`
- `status`: `pending`, `ok`, `failed`, `cancelled`, `unknown`

### ToolResult

Represents output from a tool call.

Fields:

- `tool_result_id`
- `tool_call_id`
- `event_id`
- `exit_code`
- `stdout_preview`
- `stderr_preview`
- `duration_ms`
- `status`
- `error_type`
- `output_hash`

### FileRef

Represents a file or path mentioned, read, patched, tested, or generated.

Fields:

- `file_ref_id`
- `event_id`
- `path`
- `path_role`: `read`, `write`, `patch`, `test`, `error`, `mention`, `generated`
- `line_start`, `line_end`
- `language`
- `operation`

### TestRef

Represents a test/build/lint/check signal.

Fields:

- `test_ref_id`
- `event_id`
- `tool_call_id`
- `command`
- `framework`: `pytest`, `unittest`, `npm`, `vite`, `ruff`, `mypy`, `unknown`
- `status`: `passed`, `failed`, `skipped`, `unknown`
- `failure_count`
- `summary`

### ErrorRef

Represents an error signal.

Fields:

- `error_ref_id`
- `event_id`
- `tool_call_id`
- `error_type`: `test_failure`, `runtime_exception`, `sandbox`, `network`, `permission`, `json_parse`, `timeout`, `not_found`, `unknown`
- `message`
- `stack_preview`
- `is_recovered`

### UserCorrection

Represents a user correction or clarification.

Fields:

- `correction_id`
- `event_id`
- `turn_id`
- `correction_type`: `wrong_goal`, `wrong_file`, `bad_result`, `scope_change`, `missed_requirement`, `style_feedback`, `other`
- `target_event_ids`
- `summary`

### Edge

Represents lineage between graph nodes.

Fields:

- `edge_id`
- `from_type`, `from_id`
- `to_type`, `to_id`
- `relation`: `derived_from`, `responds_to`, `calls`, `returns`, `mentions`, `patches`, `tests`, `fails`, `corrects`, `supports`, `supersedes`
- `confidence`
- `created_at`

## Source Ref Format

Every evidence-bearing object needs a stable ref:

```text
{source_type}:{session_id}:turn_{turn_index}:event_{event_index}
{source_type}:{session_id}:turn_{turn_index}:tool_{tool_call_index}
{source_type}:{session_id}:turn_{turn_index}:file_{file_ref_index}
```

Examples:

```text
codex:abc123:turn_4:event_9
cursor:chat-87:turn_2:tool_1
claude-code:2026-06-13-01:turn_8:file_3
```

Source refs are what reports, evidence packs, validations, and skills cite.
Raw DB integer IDs are internal and should not appear as public evidence refs.

## Normalization Pipeline

1. Parse source-specific records into `RawArtifact` and `RawRecord`.
2. Identify or create `Session`.
3. Assign records into `Turn`.
4. Split records into semantic `Event`.
5. Extract `ToolCall`, `ToolResult`, `FileRef`, `TestRef`, `ErrorRef`, and `UserCorrection`.
6. Create `Edge` records for provenance and causality.
7. Compute deterministic facts from graph tables.
8. Build phase-based `Episode`.
9. Build `EvidencePack` from source refs and graph facts.
10. Pass only evidence packs to LLM workflow.

## Chat Record Chain

The chain from raw chat record to report must be inspectable:

```text
raw_record.raw_record_id
  -> event.raw_record_ids
  -> edge(derived_from)
  -> evidence_pack.source_refs
  -> issue.evidence_refs
  -> cluster.evidence_refs
  -> validation.evidence_refs
  -> report cluster card
  -> skill_candidate evidence
```

This makes every LLM claim auditable.

## API Contract

### `GET /transcripts/:sessionId/graph`

Returns the structured graph for one session.

Response:

```json
{
  "ok": true,
  "session": {},
  "turns": [],
  "events": [],
  "tool_calls": [],
  "tool_results": [],
  "file_refs": [],
  "test_refs": [],
  "error_refs": [],
  "user_corrections": [],
  "edges": []
}
```

### `GET /transcripts/:sessionId/lineage?ref=...`

Returns upstream and downstream lineage for a source ref.

Response:

```json
{
  "ok": true,
  "ref": "codex:abc123:turn_4:event_9",
  "upstream": [],
  "downstream": [],
  "evidence": []
}
```

### `POST /transcripts/:sessionId/rebuild-graph`

Rebuilds normalized graph records from raw records for one session.

Use this when parser versions change.

## Storage Plan

Keep existing `sessions` and `events` for compatibility.
Add graph tables incrementally:

- `raw_artifacts`
- `raw_records`
- `turns`
- `normalized_events`
- `tool_calls`
- `tool_results`
- `file_refs`
- `test_refs`
- `error_refs`
- `user_corrections`
- `transcript_edges`
- `normalization_runs`

Existing workflow can keep working while the graph layer is introduced.
Once stable, `analysis_workflow.py` should consume `normalized_events` instead of deriving them in memory from `TranscriptEvent`.

## Versioning

Store parser and graph versions:

- `parser_version`: source importer parser version
- `graph_schema_version`: normalized graph contract version
- `normalizer_version`: rules used to produce phases and refs

When any version changes, graph rebuild is allowed without re-importing raw files.

## Testing Strategy

- Unit tests for each source parser preserving raw record IDs.
- Unit tests for turn assignment and phase classification.
- Unit tests for tool call/result pairing.
- Unit tests for file/test/error/user correction extraction.
- Golden tests for Codex, Cursor, and Claude Code sample sessions.
- Lineage tests proving raw record -> issue -> cluster -> report references.
- Performance tests for large JSONL sessions and Cursor SQLite stores.

## Boundaries

- Always preserve raw record provenance.
- Always keep source refs stable across rebuilds when raw content is unchanged.
- Always validate source-specific parser output before graph insertion.
- Never let LLM write graph facts directly.
- Never use raw full chat text as LLM input.
- Never generate `SKILL.md` from a cluster without validated source refs and human confirmation.

## Success Criteria

- A session can be displayed as a structured graph with turns, events, tools, files, tests, errors, and corrections.
- Every report claim can be traced back to raw source records.
- Re-importing unchanged files does not duplicate graph records.
- Parser changes can rebuild graph records from raw records.
- LLM workflow consumes evidence packs built from graph refs.
